# Аудит плагина «teacher-assistant» — 26.04.2026

**Аудитор:** Claude (Opus 4.7)
**Версия плагина:** 1.0.0
**Метод:** статический анализ кода + сверка нормативной базы Минпросвещения РФ через web (publication.pravo.gov.ru, КонсультантПлюс, edsoo.ru, fipi.ru) + прогон тестов и валидаций.

## TL;DR

Плагин технически работоспособен (60/60 тестов проходят, typecheck чистый, бандл собирается, валидация ФГОС-данных без ошибок), но имел **существенные нормативные неточности** и несколько **архитектурных шероховатостей**, унаследованных от спешной первой сборки. В рамках этого аудита применены безопасные фиксы (см. раздел «Что уже исправлено»). Остающиеся пункты вынесены в раздел «План улучшений по приоритетам».

**Итоговая оценка после фиксов:** 8.5/10 (для production-готовности нужно отработать пункты P1).

## 3. Технический аудит — обнаруженные дефекты

### Конфигурационная архитектура

В корне плагина соседствуют **6** манифестных файлов с пересекающимися полями: `plugin.json` (root), `.claude-plugin/plugin.json`, `.claude-plugin/marketplace.json`, `manifest.json` (MCPB/Claude Desktop), `.mcp.json`, `server.json` (MCP Registry), `package.json`. Это нормально для multi-channel дистрибуции (Claude Code plugin + MCP Registry + npm + MCPB), но:

| Файл | Реально читается кем | Что было неконсистентно |
|---|---|---|
| `.claude-plugin/plugin.json` | Claude Code | `${CLAUDE_PLUGIN_ROOT}/mcp-server/dist/bundle.js` — **корректно** |
| `.mcp.json` | Claude Code (legacy/standalone MCP config) | `mcp-server/dist/bundle.js` (относительный) — **исправлено** на `${CLAUDE_PLUGIN_ROOT}/...` |
| `manifest.json` | Claude Desktop (MCPB-bundle) | `mcp-server/dist/bundle.js` — **корректно** для MCPB-формата (пути относительно бандла), не трогали |
| `server.json` | MCP Registry | npm-имя `@knottasoft/teacher-assistant-mcp` — корректно |
| `package.json` (root) | npm | `@knottasoft/teacher-assistant` — корректно |

### Hooks (`hooks/hooks.json`) — БЫЛО

```json
{
  "type": "command",
  "command": "if echo \"$TOOL_INPUT\" | grep -q '\\.md\"'; then echo 'CHECK'; else echo 'SKIP'; fi"
}
```

`$TOOL_INPUT` — **не существующая переменная окружения** в API Claude Code hooks. Hook payload приходит через **stdin как JSON**: `{tool_name, tool_input: {file_path, content}, ...}`. Значит, прежняя проверка всегда возвращала `SKIP`, но всё равно дёргала следующий `prompt`-hook (LLM-вызов на каждый Write).

**Сделано:** заменено на полноценный node-скрипт `hooks/check-teaching-doc.mjs`, который читает stdin, фильтрует только `.md` в `user-data/` (или с ключевыми словами), и при отсутствии класса/предмета/привязки к ФГОС возвращает `additionalContext` через `hookSpecificOutput` (не блокируя запись). Дорогостоящий `prompt`-hook на каждый Write **удалён**. Поведение протестировано вручную.

### MCP-инструменты

| Инструмент | Дефект | Статус |
|---|---|---|
| `export-docx.ts:createTable` | По старому REVIEW.md возвращал пустой Paragraph вместо Table | **Уже исправлено** в коммите #3 — сейчас возвращает `[table]`, тесты проверяют |
| `import-template.ts` | Path traversal: расширение бралось из непроверенного `template_path`, плюс использовался синхронный fs | **Исправлено**: `extname` валидируется по whitelist (`.md`, `.markdown`, `.txt`), путь резолвится в абсолютный, добавлена проверка что target внутри `USER_TEMPLATES_DIR` |
| `fgos-lookup.ts` | При отсутствии совпадений возвращал `gradeData.sections` целиком — мог взорвать контекст | **Исправлено**: при пустом результате возвращается компактная подсказка с перечнем доступных разделов и тем |
| `grade-analytics.ts` | `readFileSync` в async-хендлере (по REVIEW.md) | Не трогали — отложено в P2 |

### Skills

18 SKILL.md, все с корректным YAML frontmatter, проходят CI-валидацию. Замечания из старого REVIEW.md остаются открытыми:
- Используется нестандартный формат `@templates/`, `@examples/` для ссылок на supporting files (правильно — Markdown-ссылки)
- `disable-model-invocation` не выставлен явно — при 18 skills возможен перебор бюджета описаний

### npm vulnerabilities

`npm audit` в `mcp-server/`: 8 уязвимостей (1 moderate, 7 high), все в транзитивных зависимостях (`@hono/node-server` через MCP SDK). Прямого риска для нашего stdio-транспорта нет (мы не используем serveStatic), но стоит закрыть обновлением SDK.
